Настройка новой среды разработки в теории должна занимать пару команд и немного ожидания. На практике часто выглядит так:

```
$ docker pull pytorch/pytorch:latest
Pulling fs layer
Downloading [==>                     ]  48.2MB/1.2GB
```

Прогресс замирает на каком-то слое, а через несколько минут появляется `net/http: TLS handshake timeout`. Или загрузка весов модели с Hugging Face обрывается на 60% с ошибкой `Connection reset`, и приходится начинать заново. Или `npm install` просто зависает, а после Ctrl+C и повторного запуска вдруг всё получается.

Общее у этих проблем одно: **дело не в коде, а в нестабильности сети при загрузке ресурсов из-за рубежа**, и такие сбои легко упустить из виду, потому что обычно их просто "перезапускают", а не воспринимают как системную проблему.

## Почему загрузка зависимостей так чувствительна к сети

Большинство зависимостей современной среды разработки размещены за рубежом:

- **Docker Hub** (`registry-1.docker.io`) — источник официальных и большинства community-образов
- **Hugging Face** (`huggingface.co` / `cdn-lfs.huggingface.co`) — веса моделей и датасеты, файлы от сотен мегабайт до десятков гигабайт
- **npm registry** (`registry.npmjs.org`) и **PyPI** (`pypi.org`) — не все пакеты вовремя попадают в локальные зеркала
- **Go modules** (`proxy.golang.org`), **Cargo** (`crates.io`)

Общая черта этих загрузок: **большой объём файлов и долгое удержание соединения**. Загрузка слоя образа на 2GB или весов модели на десятки гигабайт может занимать от нескольких минут до десятков минут — и всё это время сеть должна оставаться стабильной. Любой серьёзный обрыв из-за потери пакетов может обнулить уже загруженный прогресс.

## Разбор типичных ситуаций

**Docker pull зависает на слое**: образы загружаются по слоям, каждый — отдельный HTTP-запрос. При перегрузке международного канала скорость может упасть до нескольких KB/s, что выглядит как зависание, а на деле — крайне медленная загрузка, которая в итоге упирается в таймаут клиента.

**Загрузка моделей с Hugging Face обрывается и требует докачки**: теоретически поддерживается докачка, но при частых обрывах реальная эффективность падает в разы — для моделей на десятки гигабайт итоговое время может быть в 5–10 раз выше теоретического.

**npm/pip зависают или выдают `ETIMEDOUT`**: небольшие пакеты обычно устанавливаются мгновенно, но если пакет отсутствует в локальном зеркале и приходится идти напрямую к официальному источнику, нестабильность канала сразу становится заметной.

## Сравнение измерений: скорость загрузки типичных зависимостей

Условия теста: домашняя сеть 300 Мбит, будний день, вечерний час пик.

| Загружаемый ресурс | Прямое соединение | Обычный прокси | NasaCode IEPL |
|---------------------|--------------------|-----------------|----------------|
| Docker-образ (pytorch/pytorch, ~6GB) | частые обрывы, в среднем 21 мин | 8 мин 30 с | 2 мин 30 с |
| Модель Hugging Face (7B, ~14GB) | частые обрывы, 40+ мин | 17 мин | 5 мин |
| npm install (новый немиророванный пакет) | таймаут в 33% случаев | таймаут в 6% случаев | таймаут в 0,4% случаев |
| pip install (официальный источник) | 7,9 с | 3,0 с | 0,9 с |
| Средняя скорость загрузки больших файлов | 0,6–2 МБ/с, нестабильно | 6–8 МБ/с | 25–35 МБ/с, стабильно |

Загрузка модели Hugging Face сокращается с 40+ минут (с частыми обрывами) до примерно 5 минут — критическая разница для тех, кто часто переключается между моделями в экспериментах.

## Почему IEPL-линия так эффективна для больших файлов

**Выделенная полоса пропускания устраняет перегрузку в часы пик**: IEPL — это выделенная физическая линия между операторами, не проходящая через публичные точки обмена трафиком, поэтому скорость не падает из-за других пользователей сети.

**Низкий процент потерь пакетов снижает издержки на повторную передачу**: при потере пакетов TCP снижает скорость и восстанавливает её постепенно. Потери на IEPL обычно ниже 0,05%, что позволяет держать высокую пропускную способность соединения.

**Стабильность соединения снижает частоту обрывов**: меньше ситуаций, когда загрузка на 80% внезапно обрывается и приходится докачивать или начинать заново.

## Настройка

1. Скачайте клиент [NasaCode](https://www.nasacode.com/ru/download) (Windows / macOS / Linux)
2. Выберите узел с пометкой **IEPL** в списке серверов — рекомендуется для сценариев с большими файлами
3. Включите **TUN-режим** — команды `docker pull`, `pip install`, `huggingface-cli download` автоматически пойдут через ускоренный канал без дополнительной настройки

При использовании режима системного прокси некоторые инструменты требуют отдельной настройки:

```bash
export HTTPS_PROXY=http://127.0.0.1:7890
export HTTP_PROXY=http://127.0.0.1:7890

pip install --proxy http://127.0.0.1:7890 package-name

npm config set proxy http://127.0.0.1:7890
npm config set https-proxy http://127.0.0.1:7890
```
